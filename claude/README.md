# Claude tooling notes

Scripts and notes saved while working on this codebase, kept here for future reuse rather than thrown away.

## Files

- **inspect-step.test.mts** â€” vitest test that simulates the reducer up to any step (same logic as `mapReducer.ts`'s `directStep`) and reports the resulting view, which text labels are on-screen, and which country polygons intersect the rendered viewport (with % coverage). Useful for a first-pass survey of "what's visible at step N." Run with `STEP=<n> npx vitest run claude/inspect-step.test.mts`. Don't trust the "point" it derives for label placement, though â€” see **find-label-point.test.mts** below.
- **find-label-point.test.mts** â€” the tool that actually works for placing labels. For named countries, searches for a point that's verifiably inside the country's *real, unclipped* polygon with good clearance from its border in all four directions. Run with `STEP=<n> COUNTRIES=Italy,RomaniaFinal npx vitest run claude/find-label-point.test.mts` (see the file header for more env vars, including `PREFER_CENTER` for labels that should hug the view's center instead of maximizing border clearance).
- **screenshot-step.mjs** â€” Playwright script that jumps straight to a step via the step buttons and saves a screenshot. Run with `node claude/screenshot-step.mjs <step>` against a running dev server. Always sanity-check label placements with this â€” geometry checks alone missed the text-anchor clipping issue noted below.
- **record-video.mjs** â€” Playwright script that clicks "NEXT" through the whole animation and records the session as a `.webm` video in `video-output/`.

The Playwright scripts need `playwright` installed (`npm install -D playwright && npx playwright install chromium`) and a dev server running (`npx vite --port 5183 --strictPort`). Neither is a permanent dependency of this repo.

## Things learned about the codebase

- **The app is now pinned to a 16:9 viewport, by design.** `MapAnimation.tsx`'s `viewBox` is always exactly `2` units wide (`${x-1} ${y-height/2} 2 ${height}`), with only `height` varying by zoom â€” under the SVG default `xMidYMid meet` (no `preserveAspectRatio` override), height is always the constraining dimension, so the visible longitude range depends entirely on the *element's* pixel aspect ratio: `effectiveWidth = height * (elementPixelWidth / elementPixelHeight)`. Originally the SVG just filled the raw browser window, so resizing the window changed what was visible â€” not just the zoom level. `App.css`'s `.viewport`/`.container` now constrain the SVG's actual rendered box to a fixed 16:9 (`width: min(100vw, 100vh * 16/9)`, `height: min(100vh, 100vw * 9/16)`, centered, with letterbox bars otherwise), so `elementPixelWidth/elementPixelHeight` is always 16/9 regardless of window size â€” the map only ever scales uniformly, never crops differently. If you ever need to know "what's on screen" for label placement, you can now assume 16:9 unconditionally instead of guessing the user's window shape (`inspect-step.test.mts`'s `ASPECT` env var should normally just be left at its 16:9 default now).
- **`directStep` (in `mapReducer.ts`) is the source of truth for "what does state look like at step N"** â€” it replays every transition from 0 to N. The step-number buttons in the UI call this, so jumping to a step via those buttons is equivalent to (and much faster than) clicking NEXT repeatedly.
- **`WW1.tsx`'s `initialState` and `transitions` are exported** (as of this work) specifically so tooling like `inspect-step.test.mts` can import and simulate them without spinning up a browser.
- Labels (`MapText`) fading out don't get repositioned â€” `TextFadeIn` on an existing id only touches opacity, not coordinates (see `mapReducer.ts`'s `TextFadeIn` case). To relocate a currently-visible label, use `textMove`, not another `textFadeIn`.
- **The step-button locator must be scoped to its actual parent.** The "Start"/numbered step buttons live in a `<div>` nested inside `.controls`, separate from the `NEXT` button. A locator like `.controls button:not(:nth-child(1)):not(:nth-child(2))` looks like it skips "NEXT" and "Start", but `:nth-child` is evaluated per-element against *its own* siblings â€” so it also (wrongly) excludes the step-0 button, which is `nth-child(2)` of the inner div. This silently produced an off-by-one (e.g. `record-video.mjs` would click NEXT one time too few, stopping short of the final step). Use `.controls > div > button:not(:first-child)` instead, which is properly scoped to the inner div.
- **Placing labels took three attempts to get right â€” here's the whole story, so it isn't repeated:**
  1. `turf.centroid()` of a country's polygon is just the average of all vertices. For concave or multi-part shapes (Italy's boot + islands) the average can land in the sea or in a neighboring country. (`Italy` ended up in the Adriatic, `Bulgaria`/`Romania` landed across their borders.)
  2. Switching to `turf.pointOnFeature()` on the polygon **intersected with the viewport bbox first** seemed to fix it â€” `pointOnFeature` guarantees a point that lies on its input geometry. But it's only as good as that input geometry, and `turf.intersect()` against a bbox can produce a degenerate/wrong fragment for a low-resolution, multi-ring country shape (this repo's geojson is explicitly low-res â€” see the "Fix fragments for low-res map" commit). The result *passed* `pointOnFeature`'s guarantee while still landing outside the country's real, unclipped polygon. Checking `turf.booleanPointInPolygon(point, country)` against the **actual full country geometry** (no bbox intersection) caught this immediately â€” `(13.4, 44)` for Italy tested `false`, despite looking fine on paper. Always verify against the real polygon, not a derived/clipped one.
  3. Even once a point is confirmed inside the real polygon, it can be right at the boundary with almost no clearance â€” enough to make a multi-character label visually cross a border or coastline. `find-label-point.test.mts` fixes this by grid-searching for a point with margin (clearance in all 4 cardinal directions) above some threshold, optionally preferring the point closest to the view's center instead of the most clearance (for `Austria-Hungary`, which needed to read as "near the middle of the screen" while *also* staying off the Serbia border â€” there's a real tension between those two goals, so don't over-optimize for one at the total expense of the other).
- **A label can clip off-screen even at a verified on-land, in-frame coordinate.** SVG `<text>` is center-anchored by default, so it extends in both directions from its coordinate â€” a point just barely inside the visible viewport can still get its label clipped by the screen edge. Pick points with a buffer from the viewport edge, not just from the country's own border, and reconfirm with an actual screenshot (`screenshot-step.mjs`) â€” geometry checks alone won't catch this.
- **There's no single "right" viewport to test against.** This app's visible area depends on the browser window's aspect ratio (see the viewBox note above), and that varies per user. `find-label-point.test.mts` defaults its search box to the intersection of a ~16:9 and a wider (~2.1) window's visible longitude range, which is a reasonable "probably visible for most people" compromise â€” but there's no guarantee for an arbitrary window size. When in doubt, screenshot at more than one aspect ratio (see how `screenshot-step.mjs` is invoked vs. the inline Playwright snippet used to check a custom viewport size).
- **Focus previews use a `MapHighlight` concept, parallel to `MapText`.** First attempt was a dashed "pointer line" from the summary box to the focus label(s) (a `MapLine` type) â€” functional, but it read as decoration rather than as "this is what's about to change," so it was replaced with an outline glow traced around the actual target shape instead. `mapReducer.ts` has `highlightFadeIn`/`highlightFadeOut` + a `highlightCollection` on state, mirroring `textFadeIn`/`textFadeOut`/`textCollection` exactly (same opacity-fade reducer logic, same `directStep` replay support).
  - `MapHighlight` has an `id` plus an *optional* `coordinates`. When `coordinates` is omitted, `MapAnimation.tsx` looks `id` up as a country name in `countries` at render time, so the outline tracks whatever that country's geometry currently is (handles `CountryReplace`/`CountryFadeIn` automatically). `countries.ts`'s `countryHighlight(countryName, svgPathProps?)` builds this kind.
  - When you need to preview an upcoming *merge* (several countries' borders dissolving into one new blob), a per-country outline is misleading â€” it draws separate boundaries (and visible internal seams between them) when the real "next step" is one combined region. For that, supply `coordinates` explicitly via `countries.ts`'s `areaHighlight(id, coordinates, svgPathProps?)`, built from a union of just the countries being merged â€” same `union(...)` helper from `utility.ts` the `*Union` files in `positionCalc/` already use.
  - **Don't assume an existing `*Union` variable is scoped the way the highlight needs â€” it's usually scoped for the merge, not the preview, and those can differ.** `russiaMiddleEastUnion` (step 11's `countryFadeIn`) unions Georgia/Azerbaijan/Armenia/Kars *with Russia's existing territory*, since that's what the final merged country needs; using it directly for the step 10 preview highlight would wrongly outline Russia too, implying Russia itself was about to change. Similarly `russiaBalticsUnion` (step 3) unions Estonia/Latvia/Lithuania (with Lithuania's Memel area carved out, since that goes to Germany much later) *plus* Russia/Finland's existing territory â€” wrong shape for a step 2 preview highlight for the same reason, and the Memel carve-out is irrelevant noise for a "these three countries are about to change" preview anyway. Both got a narrower sibling file in `positionCalc/` containing only the highlight-relevant pieces: `caucasusUnion.ts` (Georgia/Azerbaijan/Armenia/Kars, no Russia) and `balticsUnion.ts` (plain Estonia/Latvia/Lithuania, no Russia, no Memel carve-out) â€” read what an existing union actually contains before reusing it, and don't hesitate to add a new minimal file rather than force-fitting one that's scoped for a different purpose.
  - Same idea applies one level smaller: when only a *sub-region* of a country is changing hands (step 12's Alsace-Lorraine/South Jutland, annexed from France/Denmark into Germany), highlighting the whole of France or Denmark would be wrong too â€” most of each country isn't changing. `germanyFranceDenmarkUnion` already pulls in exactly these two sub-regions from static data files (`data/AlsaceLorraine.json`, `data/SouthJutland.json`), unioned with *Germany's* existing territory for the actual merge. `alsaceLorraineSouthJutlandUnion.ts` unions just those same two data files together, no country lookup at all â€” note it takes no `state` parameter, unlike its `*Union` siblings, since there's nothing in `state.countries` it needs (don't add an unused parameter just to match the sibling files' signature; `noUnusedParameters` in `tsconfig.json` will reject it anyway).
  - Rendered in `CountryHighlight.tsx` as two stacked strokes per coordinate ring (a wide faint halo + a crisp line, both gold, sharing one fade-driven opacity) â€” drawn in its own `<g>` between the countries and text layers, so the glow sits over the map but doesn't cover label text. See step 0 (highlight `Finland`), step 2 (highlight the *combined* `balticsUnion` area, not Estonia/Latvia/Lithuania individually), step 10 (highlight the *combined* `caucasusUnion` area, not Georgia/Azerbaijan/Armenia individually, and not Russia), and step 12 (highlight just the `alsaceLorraineSouthJutlandUnion` sub-regions, not all of France/Denmark) in `WW1.tsx`. General pattern: fade the highlight(s) in alongside the summary + label(s), fade them out in whichever later transition is where the actual border change happens (not necessarily the same transition where the summary itself fades out).
  - This pattern was repeated for every remaining "summary â†’ merge" pair in the storyline: steps 14/16/18/20/22/24/26/28/30/32/34/36 each fade in a highlight alongside their summary, and the matching 15/17/19/21/23/25/27/29/31/33/35/37 fade it out where the actual merge happens. Most needed a new minimal `positionCalc/*Union.ts` file scoped to just the changing pieces (`polandWestNorthUnion`, `kaliningradMemelUnion`, `czechSlovakiaUnion`, `balkanTrioUnion`, `galiciaBukovinaUnion`, `levantUnion`), with the corresponding real merge file (`germanyPolandUnion`, `germanyFinalUnion`, `ahCzechUnion`, `ahBalkansUnion`, `ahFinalUnion`, `ottomanMiddleEastUnion`) refactored to reuse it rather than duplicate the lookup/union logic â€” same "use the new union in the union that needs it" pattern as `caucasusUnion`â†’`russiaMiddleEastUnion` earlier. A few sub-region highlights (Trentino/South Tyrol, Vojvodina, ex-Romania/Transylvania) are each just one static data file already used by exactly one existing `*Union`, with no other pieces to combine â€” for those, the data file (e.g. `data/TrentinoSouthTyrol.json`) is imported straight into `WW1.tsx` and passed directly to `areaHighlight`, no new `positionCalc/` file needed; don't add a wrapper file just for uniformity when there's nothing to compute.
  - **For step 34's Serbia/Kosovo/North Macedonia preview, the raw `union()` of the three countries was the wrong shape** â€” it included pieces (Vojvodina, slivers along the Macedonia/Bulgaria border) that had already been ceded away to Austria-Hungary/Bulgaria in *earlier* steps, so the highlight visibly bled into neighboring countries instead of stopping at Serbia's actual edge. The fix: highlight `serbiaFinalUnion` directly (already computed for step 35's real `countryFadeIn`) instead of an intermediate union â€” i.e. "use the shape the result will actually have," not "union of the input countries," whenever a `difference(...)` cleanup step separates the two. The intermediate `serbiaKosovoMacedoniaUnion.ts` ended up with no remaining callers once `WW1.tsx` stopped needing it directly, so it got inlined back into `serbiaFinal.ts` (the one place that still needed the union-before-difference step) and deleted, rather than kept around as a single-caller indirection.
  - The Balkan Wars step (32/33) started as a `balkanWarsChangesUnion.ts` unioning every individually-changing piece (Bulgaria gains several Macedonia/Thrace pieces, loses South Dobruja and West Thrace, spread across three different result countries) â€” geometrically accurate, but per explicit follow-up feedback the user actually wanted the simpler `countryHighlight('Bulgaria')` (the whole existing country, like the very first highlight examples) rather than a scattered multi-piece preview. Deleted the now-unused file rather than leaving dead code around. Worth remembering: "highlight everything that changes" and "highlight the country the summary names" aren't always the same thing, and the latter can be the better read even when it's geometrically less precise.
  - **When you need to check whether a highlight's shape is actually right, verify with geometry, not by eyeballing a small screenshot crop.** A zoomed crop of the ex-Romania/Transylvania highlight looked like it might be overlapping already-existing Austria-Hungary territory â€” but `turf.booleanPointInPolygon`/`turf.intersect` against the actual country polygons at that step confirmed 0 kmÂ˛ overlap with Austria-Hungary and full overlap with Romania. The screenshot just didn't have enough contrast/resolution at that crop to read correctly; trust the math over a quick visual scan when something looks ambiguous.
